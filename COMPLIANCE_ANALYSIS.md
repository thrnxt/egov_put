## 📋 Требования из Приказа №1187 и их реализация

### **Пункт 3: Функционал проверки подлинности ЭЦП**

#### Требование 3.1: Проверка ЭЦП в электронном документе
**Статус:** ✅ **РЕАЛИЗОВАНО** (через NCANode)

**Реализация:**
- Метод: `validateSignatureViaNcaNode()` в `SignService.java`
- Поддерживаемые методы подписи:
  - `CMS_WITH_DATA` - CMS подпись с данными
  - `CMS_SIGN_ONLY` - только CMS подпись
  - `XML` - XML подпись
  - `SIGN_BYTES_ARRAY` - подпись массива байтов

**Код:**
```java
// SignService.java, строки 232-271
private boolean validateSignatureViaNcaNode(Api2Response signedData) {
    for (Api2Response.DocumentToSign doc : signedData.documentsToSign()) {
        String signMethod = doc.signMethod() != null ? doc.signMethod() : signedData.signMethod();
        boolean isValid = false;
        
        switch (signMethod) {
            case "CMS_WITH_DATA":
            case "CMS_SIGN_ONLY":
                isValid = validateCmsSignature(doc);
                break;
            case "XML":
                isValid = validateXmlSignature(doc);
                break;
            // ...
        }
    }
}
```

### **Пункт 5: Схема проверки ЭЦП**

**Требование 5.1:** Дешифрование хэша сообщения с помощью открытого ключа
**Статус:** ✅ **РЕАЛИЗОВАНО** (через NCANode)

**Требование 5.2:** Вычисление контрольной суммы оригинального сообщения
**Статус:** ✅ **РЕАЛИЗОВАНО** (через NCANode)

**Требование 5.3:** Сверка контрольных сумм
**Статус:** ✅ **РЕАЛИЗОВАНО** (через NCANode)

**Реализация:**
```java
// SignService.java, строки 273-295
private boolean validateCmsSignature(Api2Response.DocumentToSign doc) {
    String requestBody = String.format(
        "{\"cms\": \"%s\", \"checkOcsp\": true, \"checkCrl\": true}",
        cmsBase64
    );
    return callNcanodeVerify("/cms/verify", requestBody);
}
```

**Результат:** NCANode возвращает `{"valid": true/false}` после выполнения всех проверок

### **Пункт 6: Проверка регистрационного свидетельства**

При положительном результате проверки ЭЦП выполняются следующие проверки:

#### **6.1) Проверка срока действия регистрационного свидетельства**
**Статус:** ✅ **РЕАЛИЗОВАНО** (через NCANode)

**Детали:**
- NCANode проверяет поля `notBefore` и `notAfter` сертификата
- Проверяет всю цепочку до корневого сертификата УЦ
- Проверяет промежуточные сертификаты

**Логирование:**
```java
// SignService.java, строки 352-359
if (jsonNode.has("certInfo")) {
    log.debug("  - Valid from: {}", certInfo.get("notBefore").asText());
    log.debug("  - Valid to: {}", certInfo.get("notAfter").asText());
}
```

---

#### **6.2) Проверка на отозванность (аннулирование)**
**Статус:** ✅ **РЕАЛИЗОВАНО** (через NCANode с явными параметрами)

**Поддерживаемые методы согласно Приказу:**

**A. Проверка через СОРС (Список отозванных регистрационных свидетельств)**
- **Параметр:** `"checkCrl": true`
- **Статус:** ✅ Включен в запросе

**B. Онлайн проверка через OCSP (On-line Certificate Status Protocol)**
- **Параметр:** `"checkOcsp": true`
- **Статус:** ✅ Включен в запросе

**C. Проверка через дополнительный СОРС**
- **Статус:** ✅ NCANode поддерживает автоматически при наличии дополнительных CRL

**Реализация:**
```java
// SignService.java, строки 290-293
String requestBody = String.format(
    "{\"cms\": \"%s\", \"checkOcsp\": true, \"checkCrl\": true}",
    cmsBase64
);
```

**Логирование:**
```java
// SignService.java, строки 365-367
if (jsonNode.has("revoked") && jsonNode.get("revoked").asBoolean()) {
    log.error("Certificate is revoked (СОРС/OCSP check failed)");
}
```

---

#### **6.3) Проверка области использования ЭЦП (KeyUsage)**
**Статус:** ✅ **РЕАЛИЗОВАНО** (через NCANode)

**Детали:**
- NCANode проверяет поле `KeyUsage` в сертификате
- Проверяемые значения:
  - **"Цифровая подпись" + "Неотрекаемость"** → сертификат для ЭЦП
  - **"Цифровая подпись" + "Шифрование ключей"** → сертификат для аутентификации

**Примечание:** NCANode автоматически проверяет соответствие `KeyUsage` для операции подписи

---

#### **6.4) Проверка политики сертификата**
**Статус:** ⚠️ **ЧАСТИЧНО РЕАЛИЗОВАНО** (через NCANode, но без дополнительной проверки в коде)

**Что проверяет NCANode:**
- Наличие поля `certificatePolicies` в сертификате
- Соответствие политике УЦ

**Что нужно добавить (при необходимости):**
- Проверка разрешенных способов использования сертификата
- Ограничение использования по информационным системам
- **Рекомендация:** Добавить проверку в бизнес-логике при необходимости

**Пример дополнительной проверки (опционально):**
```java
// Если нужна проверка конкретной политики:
if (jsonNode.has("policy")) {
    String policy = jsonNode.get("policy").asText();
    if (!isAllowedPolicy(policy)) {
        log.error("Certificate policy not allowed for this system");
        return false;
    }
}
```

---

#### **6.5) Проверка цепочки сертификатов**
**Статус:** ✅ **РЕАЛИЗОВАНО** (через NCANode)

**Детали:**
- NCANode строит цепочку от проверяемого сертификата до корневого сертификата НУЦ РК
- Проверяет промежуточные сертификаты УЦ
- Проверяет подписи на каждом уровне цепочки
- Использует доверенные корневые сертификаты НУЦ РК

**Логирование:**
```java
// SignService.java, строки 354-356
log.debug("  - Subject: {}", certInfo.get("subject").asText());
log.debug("  - Issuer: {}", certInfo.get("issuer").asText());
```

---

#### **6.6) Проверка метки времени**
**Статус:** ⚠️ **НЕ РЕАЛИЗОВАНО** (требуется только для документов долговременного хранения)

**Из Приказа:**
> "Проверка квитанции метки времени осуществляется для электронных документов долговременного хранения."

**Рекомендации:**
- Если ваша система предназначена для долговременного хранения, необходимо:
  1. Получать метку времени при подписании (TSP - Time Stamp Protocol)
  2. Проверять метку времени при валидации

**Пример добавления поддержки TSP (при необходимости):**
```java
// Добавить в validateCmsSignature:
String requestBody = String.format(
    "{\"cms\": \"%s\", \"checkOcsp\": true, \"checkCrl\": true, \"verifyTimestamp\": true}",
    cmsBase64
);
```

**Статус:** ⚠️ **ОПЦИОНАЛЬНО** - зависит от требований к долговременному хранению

#### **6.7) Проверка полномочий лица, подписавшего документ**
**Статус:** ❌ **НЕ РЕАЛИЗОВАНО** (требует бизнес-логики)

**Из Приказа:**
> "Механизмы проверки полномочий осуществляются информационной системой. Проверка полномочий осуществляется при наличии информации об этом в регистрационном свидетельстве."

**Что нужно добавить:**
Проверка полномочий - это бизнес-логика вашего приложения. Примеры:
- Проверка, что подписывающий имеет право подписывать данный тип документов
- Проверка роли/должности подписывающего
- Проверка принадлежности к организации

**Рекомендация для реализации:**
```java
// Добавить в SignService.java после успешной валидации подписи:
private boolean checkSignerAuthority(JsonNode certInfo, Api2Response signedData) {
    // Извлечь данные о подписывающем из сертификата
    String subjectDN = certInfo.get("subject").asText();
    String organizationBIN = extractBIN(subjectDN);
    
    // Проверить полномочия в вашей бизнес-логике
    // Например, проверить в базе данных или через внешний сервис
    boolean hasAuthority = checkAuthorityInDatabase(organizationBIN, signedData);
    
    if (!hasAuthority) {
        log.error("Signer does not have authority to sign this document type");
        return false;
    }
    
    return true;
}
```

**Статус:** ⚠️ **ТРЕБУЕТСЯ РЕАЛИЗАЦИЯ** (зависит от бизнес-требований)
